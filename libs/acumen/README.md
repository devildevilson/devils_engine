# acumen

`libs/acumen` - экспериментальная реализация GOAP-планировщика для движка.
Библиотека не владеет игровым состоянием и не меняет мир напрямую. Ее задача -
представить текущее состояние мира компактным набором флагов, найти цепочку
символических действий, которая приводит к требуемому состоянию, и вернуть этот
план вызывающему коду.

Текущая реализация состоит из нескольких крупных частей:

- регистр метрик состояния, целей и действий;
- A* поиск по символическим переходам состояния;
- переиспользуемый scratch/аллокатор для поиска;
- точная мемоизация найденных планов.

## Модель Состояния

Состояние мира представлено типом `acumen::state`: это фиксированный
`std::bitset`. По умолчанию в нем 256 бит, размер можно изменить через
`DEVILS_ENGINE_ACUMEN_STATE_SIZE`.

Каждый бит - один GOAP-флаг. Значение бита определяется его индексом в массиве
`state_metric`, который передается в `acumen::system`.

`state_metric` сейчас связан с общей библиотекой `devils_engine::act`.
Метрика хранит имя predicate-функции, а конструктор `acumen::system` один раз
резолвит это имя через `act::registry` и кеширует типизированный указатель
`act::predicate_function*`. После этого `system::compute_state(ctx)`
перевычисляет состояние, вызывая все predicate-функции на переданном
`act::exec_context`.

То есть `acumen` сейчас ожидает, что вызывающий слой владеет:

- реальными данными актора и мира;
- `act::exec_context`, через который predicate-функции читают эти данные;
- политикой, когда и с каким бюджетом перевычислять флаги.

Идея для метрик такая: флаг может быть результатом функции или выражения вида

```text
has_effect(fear) && enemy_is_nearby
```

В текущем C++-срезе такое условие представлено predicate-функцией,
зарегистрированной в `act::registry`. Скриптовый язык и конфиги для описания
таких функций пока находятся за пределами `libs/acumen`.

## Scoped State

`acumen::scoped_state` - базовый тип для частичного описания состояния. Он
содержит:

- `handle`: значения битов;
- `mask`: какие биты имеют смысл.

Этот тип используется для требований действий, символических эффектов действий,
весов действий, валидности целей и состояния завершения цели.

Основные операции:

- `check(state)` - проверяет, что все биты под маской совпадают;
- `apply(state)` - применяет masked-изменение к состоянию;
- `compute_variance_norm(state)` - считает нормализованную дистанцию по битам
  под маской.

За счет этого действие или цель описывает только те флаги, которые ему важны,
а не полное 256-битное состояние.

## Действия

`acumen::action` содержит:

- `name`;
- `requirements` - флаги, которые должны совпасть, чтобы действие было
  доступно;
- `next_state` - символическое изменение состояния после успешного действия;
- `weight_state` - опциональные флаги для веса/приоритета действия;
- `effect` - resolved `act::effect_function*`, если функция с таким именем есть
  в `act::registry`.

Важно: планировщик не исполняет `effect`. Для A* действие - это символический
переход из одного битового состояния в другое. На выходе вызывающий код
получает список `const action*` и сам решает, как его исполнять.

В текущем `subprojects/tile_frontier` используется такая схема:

```text
имя GOAP action -> id act-функции -> act::intent -> поздняя apply-фаза
```

То есть `acumen` сейчас является слоем принятия решения, а не слоем мутации
мира. Типичный эффект действия должен генерировать небольшой intent: например
`move_intent`, flee/eat/chase-команду или другое локальное изменение актора.

## Цели

`acumen::goal` содержит:

- `name`;
- `requirements` - состояние, в котором цель считается валидной;
- `goal` - набор битов, который означает, что цель достигнута.

Список целей хранится внутри `acumen::system`, но текущая публичная точка входа
`system::decide` принимает конкретное целевое `scoped_state` через
`decide_params::goal` и отдельный `goal_id` для ключа мемоизации. Поэтому выбор
какой именно goal сейчас преследовать пока остается обязанностью вызывающего
кода.

Практическая форма цели в текущем коде - "привести ситуацию к состоянию X".
После исполнения выбранного действия реальный мир может резко измениться, и на
следующем тике актор заново перевычислит флаги и выберет новую цель или новое
действие.

## Планировщик

Планирование реализовано через `astar<acumen::astar_data>`.

Данные A* узла:

- текущий символический `scoped_state`;
- вес действия;
- указатель на действие, которое привело в этот узел.

Генерация successors проходит по всем действиям системы:

1. вычислить текущее состояние узла;
2. пропустить действие, если `requirements` не совпали;
3. посчитать вес через `weight_state`;
4. пропустить действие с почти нулевым весом;
5. применить `next_state`;
6. добавить successor в A*.

Эвристика сравнивает состояние кандидата с целевым `scoped_state`.
Стоимость перехода складывается из дистанции символического состояния и веса
действия.

Публичная точка входа - `system::decide`. Ей обязательно нужен
caller-owned `astar<astar_data>::container` в `decide_params::scratch`.
Контейнер хранит memory pool и векторы A*, поэтому ожидаемый hot-path паттерн -
один scratch-контейнер на worker thread.

`decide` возвращает полную длину плана. В переданный output span записывается
столько `action*`, сколько помещается. Если стартовое состояние уже
удовлетворяет цели, длина плана равна нулю.

## Мемоизация

`acumen::solution_cache` - опциональный точный кеш решений.

Ключ кеша состоит из:

- `goal_id`, который передает вызывающий код;
- проекции стартового состояния на `system::relevant_mask()`.

`relevant_mask()` вычисляется один раз в конструкторе системы как объединение
масок действий и целей. Биты, которые не влияют на требования, переходы, веса
или проверку цели, в ключ не попадают. Это схлопывает эквивалентные состояния и
повышает hit-rate без потери корректности.

Кеш является только ускорителем:

- hit возвращает тот же план, который нашел бы живой A*;
- miss запускает живой A*;
- если бюджет кеша заполнен, вставка пропускается, но решение все равно
  считается;
- планы длиннее `DEVILS_ENGINE_ACUMEN_MAX_PLAN` не кешируются.

`solution_cache` намеренно не синхронизирован. Ожидаемый паттерн - отдельный
кеш на поток исполнения и, при необходимости, `merge()` между кадрами для обмена
прогретыми записями.

## Что Уже Умеет

На данный момент `libs/acumen` умеет:

- хранить GOAP-состояние как фиксированный набор битов;
- перевычислять флаги через predicate-функции из `act::registry`;
- описывать действия через требования и символический next-state;
- искать план до заданного состояния через A*;
- возвращать план в caller-owned буфер без владения исполнением;
- переиспользовать A* scratch/аллокатор между решениями;
- кешировать короткие планы точной мемоизацией;
- работать в многопоточном сценарии при scratch/cache на каждый worker thread;
- интегрироваться с общей gameplay-function прослойкой `libs/act`.

Покрытие в `tests/acumen_test.cpp` проверяет:

- вычисление состояния через `act::registry`;
- двухшаговый план с prerequisite-действием;
- пустой план, когда цель уже достигнута;
- безопасную работу с маленьким output-буфером;
- ошибку загрузки при отсутствующем predicate;
- hit/miss мемоизации;
- схлопывание ключа по незначащим битам;
- поведение при заполненном бюджете кеша;
- merge нескольких кешей.

Первый крупный потребитель - `subprojects/tile_frontier/src/core/actor_simulation.cpp`.
Там `acumen` работает как cognition-арбитр для акторов: predicate-функции
описывают восприятие и внутреннее состояние, GOAP выбирает ближайшее
высокоуровневое действие, а выбранное действие превращается в `act::intent` для
поздней apply-фазы.

## Что Еще Не Сделано

Общий вид уже собран, но следующие части пока находятся вне `libs/acumen` или
не реализованы:

- описание metrics/goals/actions через конфиги;
- скриптовый язык или expression layer для predicate/effect-функций;
- встроенная политика выбора цели из списка `system::get_goals()`;
- встроенное бюджетирование перевычисления state metrics;
- глобальная или persistent-политика кеша поверх ручных per-thread кешей;
- более богатый контракт исполнения действия, кроме возврата символического
  `action*`;
- динамический размер состояния или более плотное runtime-defined
  представление флагов.

Пока границу библиотеки лучше держать узкой: `libs/acumen` отвечает за GOAP
ядро и поиск плана. Загрузка данных, скрипты, scheduling акторов и реальные
изменения мира должны жить уровнем выше.
