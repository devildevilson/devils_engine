# Стиль кода devils_engine

Этот документ описывает стиль собственного кода движка. Его цель — предсказуемый,
читаемый человеком C++ без механического бойлерплейта. Форматтер помогает соблюдать
базовые правила, но не принимает архитектурные и логические решения за автора.

Статус документа: рабочая версия. Нерешённые вопросы перечислены в конце.

## Область действия

- Правила применяются к собственному C++, CMake, скриптам, шейдерам и текстовым ресурсам.
- Код сторонних библиотек сохраняет исходное именование и оформление. Не нужно писать
  wrapper только ради сокрытия `camelCase` или `PascalCase` чужого API.
- Это относится и к импортированному в дерево коду, например `utfconv` и C-реализации
  SHA-256. Локальные изменения вокруг такого кода должны быть минимальными.
- Сгенерированные файлы и содержимое `build*`, `external`, `deps`, `exclude` не форматируются.
- При изменении существующего файла приводится в порядок затронутый логический участок.
  Глобальное переформатирование библиотеки делается отдельным коммитом.

## Файлы и инструменты

- Текст хранится в UTF-8 с окончаниями строк LF и завершающим переводом строки.
- Отступ — 2 пробела. Табуляция в собственном коде не используется.
- Хвостовые пробелы не допускаются, кроме двух пробелов осознанного Markdown hard break.
- Ограничения длины строки нет. Перенос выполняется там, где он раскрывает структуру
  длинного условия, вызова, типа или осмысленной команды.
- `.editorconfig` задаёт правила для редакторов, `.gitattributes` фиксирует LF в Git.
- `.clang-format` задаёт консервативную механическую основу. Его следует применять к
  изменённым участкам; результат нужно просмотреть вручную.
- Текущий общий стандарт языка — C++23. Переход на следующий стандарт синхронизируется
  с поддерживаемой версией Clang/NDK для Android.

## Именование

- Собственные namespaces, типы, concepts, функции, переменные, поля, enum-типы,
  enum-значения и файлы именуются в `snake_case`.
- Макросы и include guards именуются в `SCREAMING_SNAKE_CASE`. Публичный макрос получает
  префикс `DEVILS_ENGINE_` или более узкий префикс подсистемы.
- Макрос используется только там, где невозможны `constexpr`, шаблон, inline-функция,
  lambda или type alias. Вспомогательный X-macro должен быть локальным и сразу `#undef`.
- Именование чужого API сохраняется в точности. Это не считается нарушением.
- Сокращение допустимо, когда оно общеупотребимо в предметной области и не ухудшает
  поиск: `id`, `ui`, `gpu`, `rng`, `ecs`, `fsm`, `goap`.
- Имя выражает роль и единицу измерения. Для времени и размеров предпочтительны имена
  вроде `frame_time_mcs`, `byte_size`, а не неуточнённые `time` и `size`.

## Размещение кода

- Открывающая фигурная скобка остаётся на строке условия, функции, namespace, class или struct.
- Каждая самостоятельная C++-инструкция или декларация, заканчивающаяся `;`, занимает
  отдельную логическую строку. Заголовок `for` и синтаксически неделимые выражения — исключения.
- Короткие guard-конструкции допустимы в одну строку:

  ```cpp
  if (ptr == nullptr) return;
  if (!ready) continue;
  if (invalid) utils::error{}("Invalid resource");
  ```

- Если statement после `if` переносится на следующую строку, он обязательно заключается
  в `{}`. Тело `for`, range-for, `while` и `do while` всегда заключается в `{}`, даже если
  состоит из одной инструкции. Форматтер не умеет полностью выразить разницу между этими
  двумя правилами, поэтому скобки проверяются при review.
- Несколько самостоятельных действий на одной строке не допускаются.
- Логические фазы функции разделяются пустой строкой. Пустые строки не должны дробить
  один цельный шаг.
- Префиксный инкремент `++i` используется, когда старое значение не требуется.
  Постфиксный оператор остаётся в реализации самого `operator++(int)` и в выражениях,
  которым действительно нужно предыдущее значение.
- `else`, `catch` и `while` в `do` располагаются на строке закрывающей скобки.
- Указатель и ссылка визуально относятся к типу: `const resource* ptr`, `const value& value`.

## Функции и управление потоком

- Функция по возможности помещается на одном экране и решает одну понятную задачу.
- Допустимые исключения: большие инициализаторы, код с явно выделенными фазами и код,
  где подробные комментарии важнее искусственного дробления.
- Длинная функция сначала делится по смысловым шагам, а не по произвольному числу строк.
- Ранний выход предпочтителен глубокой вложенности, если он делает основной путь яснее.
- Не следует скрывать существенные побочные эффекты в неочевидных helper-функциях,
  перегруженных операторах или деструкторах.
- Для повторяющегося типобезопасного бойлерплейта предпочтительны шаблоны, concepts и
  compile-time таблицы. Ошибка шаблона должна оставаться понятной пользователю API.
- `auto` используется по умолчанию практически везде, где тип корректно выводится.
  Явный тип остаётся там, где он сам является важной частью контракта: фиксированный wire/GPU
  формат, намеренное narrowing, overload resolution или требуемый storage/lifetime.

## Аргументы, результаты и владение

- Аргументы явно разделяются на неизменяемые входные и изменяемые `in/out`.
- Скаляр, enum, указатель, handle, iterator и другой дешёво копируемый небольшой тип
  передаётся по значению. Неизменяемый входной параметр по значению записывается с
  top-level `const` и в декларации, и в определении, хотя этот qualifier не является
  частью типа функции.
- Большой или дорогой неизменяемый объект передаётся как `const T&`.
- Обязательный изменяемый `in/out` передаётся как `T&`; nullable/optional объект — как
  `T*`. Неизменяемые диапазоны и строки предпочтительно принимать как `std::span` и
  `std::string_view`.
- Жёсткий порог в 8 байт используется только как начальная эвристика. Решение должно
  учитывать ABI целевых платформ, trivial copy, ownership и aliasing. Небольшой aggregate
  может быть выгоднее передать по значению, а восьмибайтный дорогой объект — по ссылке.
- Единственный обычный результат возвращается значением. `out`-аргументы нужны для
  нескольких результатов, переиспользования storage или явного изменения объекта.
- Передача по значению с последующим move предпочтительна для аргумента, владение которым
  функция принимает безусловно.
- Raw pointer не выражает владение. Уникальное владение выражается `std::unique_ptr`;
  `std::shared_ptr` используется только при настоящем совместном lifetime.
- Общего запрета на signed/unsigned нет. Для размеров, количества и обычных индексов
  исходная точка — `size_t`; сериализация, GPU/wire formats и стабильные идентификаторы
  используют явно заданную ширину. Narrowing должен быть видимым и проверенным.

## Типы и классы

- Любое наследование записывается с явным `public`, включая `struct` и traits от
  `std::true_type`/`std::false_type`.
- Данные остаются public, если вызывающий действительно может задавать их напрямую без
  нарушения инварианта.
- Setter/getter добавляется ради инварианта, валидации, синхронизации, уведомления или
  другого побочного эффекта, а не автоматически для каждого поля.
- Для private-полей нет обязательного префикса или суффикса. Внутри одного типа и нового
  цельного модуля сохраняется локальная последовательность; переименование не делается
  только ради замены `m_value` на `value_` или обратно.
- Виртуальная функция помечается `override`; запрет дальнейшего наследования — `final`.
- Полиморфная база, удаляемая через base pointer, имеет виртуальный деструктор.
- `enum class` предпочтительнее неограниченного enum. Флаги могут иметь очевидные
  типобезопасные `|`, `&`, `^`, `~` и compound-операторы.

## Операторы и неявность

- Стандартные математические операторы перегружаются для математических типов.
- Очевидные доменные операции допустимы, например флаги через `|`/`&` и пути через `/`.
- Для операции с неочевидной стоимостью, владением или побочным эффектом используется
  именованная функция.
- Одноаргументный конструктор обычно `explicit`. Неявное преобразование разрешается,
  только если оно безопасно и естественно для домена.
- C-style cast не используется в собственном C++; выбирается подходящий C++ cast.

## Headers и зависимости

- Каждый header самодостаточен: напрямую включает все необходимые стандартные headers.
- Собственные тяжёлые зависимости минимизируются. Если для декларации достаточно forward
  declaration, полный header подключается в `.cpp`.
- Forward declaration не применяется к типу, полнота которого реально нужна: base class,
  поле по значению, inline-реализация, большинство standard-library типов.
- Сначала подключаются стандартные headers. Затем отдельными логическими группами идут
  сторонние и проектные headers; обязательной сортировки внутри групп нет.
- В публичном header не используется `using namespace`.
- Include guard строится из полного проектного пути и использует `SCREAMING_SNAKE_CASE`.
- Каждый собственный публичный header должен получить короткий общий комментарий о
  назначении, ownership/lifetime и важных ограничениях. Комментарий не пересказывает код.

## Ошибки и контракты

- Макросы assertion не нужны, если условие можно ясно записать через обычный C++:

  ```cpp
  if (resource == nullptr) utils::error{}("Resource must not be null");
  ```

- `throw`, в том числе через `utils::error`, используется для критической ошибки, после
  которой нормальное выполнение программы продолжаться не должно. `utils::error` сохраняет
  `std::source_location` и формирует диагностическое сообщение.
- Исключение не пересекает C ABI. На границе worker thread/callback критическое исключение
  должно быть залогировано и привести к предсказуемому завершению процесса, а не потеряться
  внутри подсистемы.
- Ожидаемый отказ внешних данных, нехватка очереди или отсутствие optional-ресурса не
  маскируются assertion и не бросают: API возвращает expected-подобный result/status,
  `nullptr` или явно документированный invalid id.
- Текущая проверка нового render-конфига в `graphics_base` — известное исключение из этой
  границы; её перевод на expected-подобный результат требует отдельного дизайна.
- `override` обязателен. Для `[[nodiscard]]`, `explicit` и `final` общего механического
  требования нет: они ставятся по смыслу контракта.
- `noexcept` ставится везде, где отсутствие исключения действительно гарантировано.
  Типы по возможности проектируются так, чтобы copy/move были no-throw, особенно move,
  но `noexcept` нельзя обещать, если копирование поля или вызываемая операция способны бросить.

## Комментарии, примеры и человекоориентированный API

- Комментарии объясняют причину, инвариант, lifetime, thread ownership, формат данных или
  неочевидное ограничение реализации. Они не дублируют соседнюю строку.
- Публичный инструмент сопровождается небольшим примером, когда правильное использование
  нельзя сразу понять из сигнатуры.
- Разрешены готовые helper-функции и пока неиспользуемые инструменты, если у них есть ясное
  назначение, рабочий контракт и разумная стоимость поддержки.
- Эксперимент без потребителя, со сломанным контрактом или конкурирующий с активным API
  помечается как prototype либо переносится в `exclude/` до решения.
- Код пишется прежде всего для человека: локальная очевидность важнее минимального числа
  строк и демонстрации сложности шаблонов.

## Проверка изменений

- Изменение собирается на C++23 поддерживаемым компилятором.
- Тесты выбираются пропорционально риску; исправление контракта получает focused test.
- Перед коммитом проверяются LF, trailing whitespace и `git diff --check`.
- Собственный код собирается с максимально строгим практически полезным уровнем warnings.
  Ненужное предупреждение отключается узко, с понятной причиной; новый код не оставляет
  предупреждений. Warnings стороннего кода отделяются от предупреждений проекта.
- Массовая нормализация окончаний строк или форматирование не смешиваются с изменением
  поведения без необходимости.

## Пока без отдельного жёсткого правила

- Именование private-полей кроме общего `snake_case`.
- Универсальный выбор signed/unsigned за пределами размеров, индексов и wire formats.
- Механическая обязательность `[[nodiscard]]`, `explicit` и `final`.
- Namespace closing comments.
- Отдельный полный стиль CMake/Lua/GLSL и имён ресурсов.
- Автоматическое переименование старого публичного API и сериализованных имён. Новый код
  пишется в `snake_case`; совместимость старого имени оценивается в контексте изменения.
